Why Memory Dominates ML Performance

The Memory Wall
Compute throughput has grown at ~3× per generation; DRAM bandwidth grows at ~1.4×. The gap compounds — modern ML accelerators are almost always memory-bandwidth-bound, not compute-bound.
  • Transformer attention: bandwidth scales O(n²) with sequence length
  • LLM auto-regressive decode: each token must reload all model weights
  • CNN inference: activation maps dominate SRAM pressure
  • Sparsity extraction saves FLOPs but not necessarily bytes transferred
  • Quantisation (INT8/INT4) reduces model size — primary memory-wall mitigation

On-Chip SRAM

Bitcell design · 6T / 8T / HD / LP · Voltage scaling · Yield & assist circuits

SRAM Bitcell Architectures

Standard 6T Cell
  • 2 cross-coupled inverters (4 transistors)
  • 2 access transistors (pass gates)
  • Read / write share same bitlines → contention
  • Area: ~0.021 µm² at TSMC N3E
  • Used in L1/L2 caches and SRAM cuts
8T Cell (Read-Decoupled)
  • Separate read port (2 extra transistors)
  • Eliminates read-disturb → lower Vmin
  • +20–30% area vs 6T; preferred for L1 data
  • Read SNM independent of cell ratio
HD (High Density) & LP (Low Power)
  • HD: minimum-area, for L3 / LLC / eDRAM pockets
  • LP: wider transistors, higher Vmin margin → retention SRAM, register files
  • ML accelerators use HD for scratchpad (bulk area)
Key SRAM Metrics for ML SoCs
  • Bandwidth: ports × width × frequency
  • Vmin: min voltage for retention & access (power gating depth)
  • Leakage: bitcell × array size – dominates idle power
  • Yield: 6σ targeting, assist circuits (negative BL, WL boost)

SRAM Voltage Scaling & Assist Circuits

At advanced nodes (N5, N3) device mismatch increases dramatically. Bitcell failure rate rises exponentially as supply voltage falls. Assist techniques restore margin:

Write Assist
  • Negative bitline (NBL) pull-down
  • Wordline boost (VWL > VDD)
  • VDD collapse during write
Read Assist
  • Virtual VSS (raised source)
  • Replica bitline sensing
  • Self-timed sense amplifiers
SRAM Compiler & Characterisation
Memory compilers (ARM, Cadence, TSMC internal) generate timing/power models (Liberty) and layout (GDS). Monte Carlo SPICE verifies yield targets across PVT corners. ML SoCs integrate multi-port SRAM cuts (1RW, 2RW, 1R1W) tiled to form large banks.

SECTION 02

Cache Design

Set-associativity · Replacement · Write policy · Scratchpad vs cache

Cache Fundamentals for ML SoCs

Organisation
  • Direct-mapped: simple, fast, high conflict miss rate
  • N-way set-associative: N candidates per set; LRU/PLRU/random replacement
  • Fully associative: maximum flexibility, CAM cost limits size
  • ML: 4–16 way typical for shared L2/L3
Write Policy
  • Write-back + write-allocate: dominant; minimises DRAM traffic
  • Write-through: simple coherency; used for I/O coherent agents
  • Non-temporal stores bypass cache (streaming write)
ML Workload Cache Behaviour
  • Weight matrices: large, streaming access → cache unfriendly
  • Activations: moderate reuse, fit well in scratchpad
  • Attention KV: grows with context length, spills to HBM
  • Hardware prefetch critical: stride, stream detectors
Scratchpad vs Hardware Cache
  • Scratchpad (SPM): software-managed SRAM, explicit DMA
  • Predictable latency; no tag overhead; zero conflict miss
  • Most ML NPUs use SPM + small hardware cache for CPU code
  • Google TPU: 8 MB on-chip SRAM scratchpad per chip

The ML Accelerator Memory Pyramid

Capacity vs Bandwidth Tradeoff
Moving down the pyramid: capacity grows by 100–1000× per level; bandwidth drops by 2–10×; latency increases by 5–100×.
ML mapping principle: Hot activations & accumulation buffers → SPM/L1. Weight tiles & KV-cache → L2/LLC. Full model & long-context → HBM. Cold model shards, checkpoints → CXL/NVMe.

DRAM Fundamentals

1T1C cells · Bank organisation · Timing parameters · DDR evolution

DRAM Architecture & Timing

1T1C Bitcell
  • One access transistor + storage capacitor (~10–30 fF)
  • Much smaller than SRAM — enables GB-scale density
  • Charge leaks → periodic refresh required (64ms / bank)
  • Destructive read → sense amp must rewrite
Bank & Rank Organisation
  • DRAM row = 8–16 KB; entire row activated per access
  • Row buffer (open-page policy): hit/miss/empty
  • Multiple banks allow pipelining (bank-level parallelism)
  • DDR5: 8 banks × 4 bank groups per die; 32 banks per rank

ML Scheduler Implications
Memory controllers for ML SoCs use bank-aware scheduling and row-buffer locality policies to maximise page-hit rate on weight matrix streaming patterns. DRAM ECC (on-die ECC + SECDED at interface) adds latency but is mandatory at datacenter.

DDR5 & LPDDR5/6

DDR5 — Server & Workstation
  • Data rates: 4800 – 8800 MT/s (JEDEC); 2× sub-channels (32-bit each)
  • On-die ECC: 1-bit error correction per 128-bit burst
  • Decision feedback equalisation (DFE) at receiver
  • Power management: PMIC on DIMM, 1.1 V VDD
  • ~89.6 GB/s peak per channel (DDR5-8800)
  • ML server: 8 × DDR5 channels = ~700 GB/s (Intel Gaudi 3)
LPDDR5 / LPDDR5X / LPDDR6
  • Edge AI SoCs: Apple M-series, Qualcomm Snapdragon, MediaTek Dimensity
  • LPDDR5X: 8533 MT/s, 4-channel 128-bit = ~273 GB/s (Apple M3 Ultra)
  • LPDDR6: up to 17 Gbps, CAMM2 form factor, ZQ-less calibration
  • Deep power-down, partial array self-refresh → edge efficiency
  • Package-on-package (PoP) or co-packaged with SoC die

Why not DDR for large ML SoCs?
DDR bandwidth per pin is lower than HBM/GDDR because it operates at lower data rates and uses narrower interfaces (64/72 bits per channel vs HBM's 1024-bit stack interface). PCB routing limits channel count.

GDDR7 — Graphics & Edge Inference

GDDR7 Architecture
  • Up to 36 Gbps per pin (JEDEC JESD239); PAM4 signalling replaces NRZ
  • 32-bit bus per die; multiple dies per package (256-bit typical for GPUs)
  • NVIDIA RTX 5090: 28 Gbps × 512-bit = 1.79 TB/s peak
  • On-die ECC; improved thermal design vs GDDR6X
  • Lower power per bit than GDDR6X despite higher rate
GDDR7 vs GDDR6X
  • PAM4: 2 bits/symbol vs NRZ 1 bit/symbol
  • ~1.6× bandwidth improvement per pin
  • Lower supply voltage: 1.1 V → 1.0 V
  • Better signal integrity with DFE + CTLE
When to use GDDR7
  • Desktop GPU / workstation ML
  • Cost-sensitive inference accelerators
  • Scenarios where HBM interposer cost is prohibitive
  • Lower capacity ceiling vs HBM (24–32 GB max)

High Bandwidth Memory

HBM3 · HBM3E · HBM4 · TSV stacking · Thermal challenges

HBM Architecture — TSV Stacking & Interface

3D DRAM Stacking via TSV
  • 4–16 DRAM dies stacked on base logic die; 55 µm TSV pitch (HBM3)
  • 1024-bit wide interface per stack → massive BW at low per-pin rate (~2 Gbps)
  • Eliminates PCB routing: SoC + HBM on shared silicon interposer (CoWoS-S)
  • Base die: PHY, voltage regulation, RAS (repair / redundancy)
  • Pseudo-channel architecture: 2× 512-bit channels per stack

HBM Thermal, Power & Packaging

Thermal Challenges
  • HBM stacks: 85–150 W aggregate per GPU package
  • Heat must flow laterally through silicon interposer to SoC cooler
  • Limited thermal conductance through µ-bump array
  • Junction temperature limit: 85°C (DRAM junction), throttling above
  • Thermal interface material (TIM) between die top and lid critical
HBM Interposer & Packaging
  • CoWoS-S: SoC + HBM stacks on passive Si interposer (TSMC)
  • CoWoS-L: larger interposer enables more HBM stacks (H100 SXM)
  • Samsung HBM-PIM: Processing-in-Memory (GEMV in base die)
  • SK Hynix GDDR-in-HBM: HBM4 aims for 3D integration with SoC
HBM4 — Next Generation
  • 2048-bit interface per stack (2× HBM3)
  • Target: 2 TB/s per stack, 4+ TB/s aggregate
  • Hybrid bonding to SoC die (eliminates interposer for some configs)
  • Sub-1 pJ/bit target vs ~3 pJ/bit for HBM3E
  • SK Hynix, Samsung, Micron all in qualification (2025–26)

SECTION 05

CXL Memory & Memory Pooling

CXL.mem · Type 3 devices · Memory disaggregation · Fabric switches

CXL Memory Expansion & Pooling

CXL.mem — Memory Semantic Protocol
  • CXL 2.0+: Host-managed Device Memory (HDM) — CPU can directly load/store
  • Latency: ~150–250 ns (PCIe 5/6 + CXL device latency) vs ~80 ns for local DDR5
  • Bandwidth: up to ~128 GB/s per PCIe 5.0 ×16 link; PCIe 6 doubles this
  • Type 3 device: dedicated memory expander (no compute) — MemVerge, Samsung CMM-H
  • Type 2 device: compute + memory coherent (e.g. GPU with CXL)
Memory Disaggregation for ML
  • LLM models exceed single-node HBM capacity (e.g. 1.8T param model = 3.6 TB)
  • CXL fabric switch: multiple hosts share a pool of DDR5/HBM memory modules
  • Near-memory compute in CXL device reduces data movement
  • CXL 3.1: peer-to-peer coherency across multiple CXL hosts
Software Stack
  • Linux NUMA: CXL memory appears as NUMA node; numactl placement
  • Tiered memory: hot pages on DRAM, cold pages on CXL (TPP kernel feature)
  • PyTorch / CUDA Unified Memory: transparent CXL integration in progress

Roofline Model — Visualising Memory Bottlenecks

Roofline Fundamentals
  • Arithmetic Intensity (AI): FLOPs per byte of DRAM traffic
  • Memory-bound: Performance = AI × Peak_BW   (left of ridge)
  • Compute-bound: Performance = Peak_FLOPs   (right of ridge)
  • Ridge point: AI = Peak_FLOPs / Peak_BW

Memory Optimisation Techniques

Software / Algorithmic
  • Quantisation: INT8/INT4 halves/quarters weight traffic
  • FlashAttention: fused kernel, O(n) HBM reads vs O(n²)
  • KV cache compression: grouped-query, multi-query attention
  • Paged attention (vLLM): non-contiguous KV pages
  • Tensor parallelism: shards weights across HBM stacks
  • Weight streaming: offload to CPU DDR, fetch on demand
Hardware / Microarchitecture
  • Prefetch engines: stride / stream / PC-based
  • Double buffering: overlap compute with DMA
  • Memory-side cache: near-HBM SRAM tag buffer
  • Compressed sparse format: NVIDIA A100 2:4 sparsity
  • Near-memory compute: PIM in HBM base die
  • Bank interleaving: distribute weight rows across banks
Co-design principle: Compiler (XLA, TVM, Triton) must be memory-hierarchy-aware — tile sizes, prefetch distances, and fusion decisions should be tuned to the specific SRAM capacity and HBM bandwidth of the target SoC.

Summary

On-Chip SRAM
6T/8T bitcells, assist circuits, HD vs LP flavours. ~70% die area in large ML SoCs. Vmin engineering enables power gating.
Cache & Scratchpad
Hardware cache vs software SPM tradeoffs. ML NPUs favour SPM + DMA for predictable latency and zero tag overhead.
HBM Family
1024-bit TSV interface, 3.35–4.8+ TB/s aggregate. HBM4 targets 2 TB/s/stack with hybrid bonding to SoC.
CXL Memory
Disaggregated pooling for TB-scale LLMs. ~150–250 ns latency; PCIe 6 to ~256 GB/s per link.
The Unifying Principle: ML SoC memory architecture is a bandwidth optimisation problem across five decades of capacity. The roofline model quantifies where each operator sits; the solution space spans bitcell design, packaging, compiler fusion, and algorithmic compression.
