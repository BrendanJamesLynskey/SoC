Why Specialised Hardware?
the end of free scaling
Dennard Scaling — Broken ~2006
  • Voltage no longer scales with feature size — power density rises
  • Clock frequency stalls; multi-core alleviates, not eliminates, the wall
  • Memory bandwidth grows far slower than compute — the "memory wall"
  • ILP extraction in general-purpose CPUs hits diminishing returns
Compute Demand vs Supply Gap
  • GPT-3: ~3.14 × 10²³ FLOP training
  • LLM inference: O(2·N·L) FLOP/token for N-param, L-layer model
  • Video diffusion, RL — orders of magnitude beyond NLP
  • NVIDIA A100 GPU: ~312 TFLOPS (FP16); not enough at scale
ML Workload Characteristics
  • Regularity — repeated MAC operations with fixed structure
  • Data reuse — weights reused across many inputs (batching)
  • Low precision — INT8 or FP8 often sufficient for inference
  • Sparsity — zero activations & pruned weights → skip logic
  • Parallelism — SIMD, tensor, pipeline — all exploitable
Domain-Specific Architectures (DSAs) align micro-architecture with workload structure: fixed datapaths, scratchpad memories, reduced control overhead, and dedicated precision support — yielding 10–1000× efficiency over CPUs.

Compute Primitives: MAC → GEMM → Tensor
the hierarchy of matrix operations
Scalar MAC
c += a × b
  • Single multiply-accumulate per cycle
  • CPU FPU or ALU path
  • 1 FLOP / cycle (scalar)
  • Poor reuse — reload A,B each time
SIMD / Vector
C[i] += A[i] × B[i]
  • AVX-512: 16 FP32 MACs / cycle
  • Arm SVE2: scalable vector length
  • Still 1D — limited tensor reuse
  • Compiler auto-vectorises easily
Tensor / Matrix Unit
C += A × Bᵀ
  • NVIDIA Tensor Core: 16×16×16 MMA
  • Google TPU MXU: 128×128 systolic
  • Apple AMX: 32×32 tiles per cycle
  • Maximises data reuse; bandwidth ↓↓
GEMM Tiling Strategy — Fitting the Memory Hierarchy
  • Tile into L1/register file — keep hot data in the fastest storage; amortise load cost over many MACs
  • K-dimension tiling — partial sums accumulate in accumulators, not DRAM
  • M × N output stationary — output tile stays; stream A and B rows/cols past it
  • NVIDIA cuBLAS/CUTLASS — auto-tunes tile sizes per GPU architecture (warp-level MMA)
  • Shared memory as L1 scratchpad — explicitly managed; programmer loads tiles once
  • Memory-compute overlap — async copy + Tensor Core instructions interleaved

Systolic Arrays
data flowing like a heartbeat through a 2D PE mesh
Operating Principle
  • 2D grid of Processing Elements (PEs), each performing MAC
  • Data injected from edges; flows rhythmically (systolically) across the array
  • Each PE accumulates a partial sum before passing data to its neighbour
  • Weight stationary: weights pre-loaded into PE register files; activations stream through
  • Output stationary: partial sums stay; both A and B flow (lower register pressure)
  • Row stationary (Google TPU v1): minimises total data movement energy

Dataflow Architectures
execution driven by data availability, not program counters
Dataflow vs Control-flow
  • Control-flow (von Neumann): sequential PC fetch/decode; memory bottleneck; branch misprediction overhead
  • Dataflow: instructions fire when operands are ready; implicit parallelism; no speculative execution overhead
  • ML graphs are acyclic (mostly) → perfect fit for dataflow scheduling
  • Compiler maps layer graph onto PE array dataflow schedule at compile time — zero runtime overhead
Dataflow Variants
  • Weight-stationary: weights loaded once into PE RFs; minimise weight movement energy
  • Output-stationary: partial sums accumulate locally; maximise accumulator reuse
  • Row-stationary (TPU): each PE computes one convolution row; minimises total data movement
  • No-local-reuse (NLR): all operands from global buffer — less efficient but simpler PE
Spatial Dataflow Engines
  • Cerebras WSE-3: entire wafer as one chip; 900K cores; 44 GB on-wafer SRAM — eliminates DRAM for moderate models
  • Groq LPU: deterministic single-chip SIMD; compiler schedules every cycle; zero pipeline stalls; 750 TOPS INT8
  • SambaNova SN40L: reconfigurable dataflow units (RDU); spatial compute array with streams; 2-tier DRAM+HBM
  • Tenstorrent Blackhole: mesh of RISC-V tensix cores; compiler-controlled NoC; open-source toolchain
Compiler is the hardware: spatial dataflow chips push scheduling complexity into ahead-of-time compilation. Hardware becomes a static, predictable substrate — runtime control logic shrinks dramatically, freeing area for compute.

Mixed-Precision Training (AMP)
  • Master weights: FP32 — guards gradient precision
  • Forward/backward: FP16 or BF16 — feeds Tensor Cores
  • Loss scaling: multiply loss before backward; prevents FP16 underflow
  • Accumulation: FP32 accumulator in Tensor Core MMA instruction
  • Net result: 2–3× throughput vs pure FP32, near-identical accuracy

Sparsity Exploitation
skipping zeros — architectural and algorithmic approaches
Sources of Sparsity
  • Activation sparsity: ReLU sets negative activations to zero; 50–80% sparse in practice
  • Weight pruning: iterative magnitude pruning removes small-weight connections
  • Structured pruning: remove entire filters/heads — hardware-friendly but accuracy cost
  • Unstructured pruning: arbitrary zero pattern — needs sparse HW to exploit
  • Dynamic sparsity: varies per input; harder to schedule statically
NVIDIA 2:4 Structured Sparsity
  • Exactly 2 non-zeros in every group of 4 weights
  • Stored in compressed format: 50% weight storage
  • Sparse Tensor Cores decode in hardware — 2× effective throughput
  • Metadata (2-bit indices per group) stored alongside values
  • Ampere+: hardware decompression before MMA; no software overhead
  • Accuracy: <1% loss for most vision/NLP models with retraining
Sparse Format Trade-offs
  • CSR/CSC: row/col pointer + value arrays; irregular memory; good for very sparse (>90%)
  • COO: (row, col, val) tuples; simple but high overhead at <90% sparsity
  • Block-sparse (BSR): 16×16 or 32×32 dense blocks at known positions; GPU-friendly
  • Flash Attention sparse: attends only to local & strided positions; O(N√N) compute
  • MoE routing: dynamic expert selection = structured sparsity in FFN layers
Mixture-of-Experts (MoE) — Sparsity at the Architecture Level
  • Replace dense FFN with N expert sub-networks (e.g. 8, 64, or 256)
  • Router selects top-K experts per token; only those experts execute
  • Parameter count scales with N, compute scales with K — decoupled
  • Mixtral 8×7B: 46.7B params, ~12.9B active; inference cost of 12.9B dense
  • GPT-4: rumoured MoE — explains scale vs serving cost
  • Load balancing aux loss prevents expert collapse

Memory Hierarchy in AI Accelerators
keeping compute fed — the bandwidth pyramid
On-Chip SRAM Sizing Considerations
  • Must hold input/output tiles + partial sums simultaneously
  • Larger SRAM → better roofline position; SRAM density limits die area
  • 6T SRAM: ~0.026 µm² / bit in N5; 192 MB SRAM ≈ 40 mm²
  • NVIDIA H100: 50 MB L2; Google TPU v5e: 16 MB unified buffer per chip
  • Cerebras WSE-3: 44 GB on-wafer (no DRAM bottleneck for <700B models)
Flash Attention — Algorithm-Hardware Co-design
  • Standard attention materialises full N×N score matrix — O(N²) HBM reads/writes
  • Flash Attention tiles Q,K,V into SRAM; fuses softmax + matmul kernel
  • HBM reads/writes: O(N) — linear, not quadratic
  • 2–4× wall-clock speedup; enables 100K+ context lengths
  • Flash Attention 3: async WGMMA + softmax pipelining on H100; 75% MFU

Transformer Accelerator Micro-Architecture
mapping attention and FFN layers onto silicon
Attention Bottlenecks at Inference
  • Prefill (prompt): large GEMM — compute-bound; exploits Tensor Cores well
  • Decode (autoregressive): GEMV — each token generates one row; memory-bandwidth bound
  • KV Cache: past K,V tensors stored in HBM; grows O(seq_len × layers × heads × d_head)
  • At 70B model, 100K context: KV cache ~200 GB — exceeds single GPU VRAM
KV Cache Optimisations
  • Multi-Query Attention (MQA): share single K,V head across all Q heads — 8× KV cache reduction
  • Grouped Query Attention (GQA): G shared K,V heads (Llama 3 uses G=8); balance between MHA and MQA
  • PagedAttention (vLLM): KV cache in non-contiguous pages; near-zero fragmentation; enables batching
  • Quantised KV: INT8/FP8 KV cache; 2× capacity increase at minor quality cost
Dedicated Transformer Engine (NVIDIA)
  • Hopper H100: Transformer Engine with FP8 training support
  • Per-tensor dynamic scaling maintains numerical stability
  • Async WGMMA instructions: Tensor Core + shared mem overlap
  • TMA (Tensor Memory Accelerator): hardware async data copy engine; overlaps compute with memory
  • 4× H100 Tensor Core throughput vs A100 in FP8 mode
Prefill / Decode Disaggregation
  • Prefill is compute-bound; decode is memory-bound → different optimal hardware
  • Disaggregated serving: separate GPU pools for prefill vs decode (Splitwise, DistServe)
  • Prefill pool: large A/H100; decode pool: cheaper memory-bandwidth GPUs
  • Reduces total cost of serving by 30–40% at scale

Quantisation for Inference
reducing bit-width without sacrificing accuracy
Post-Training Quantisation (PTQ)
  • Symmetric: scale = max(|W|) / 127; zero-point = 0; preferred for weights
  • Asymmetric: separate scale + zero-point; better for asymmetric activation distributions
  • Per-tensor: one scale for whole tensor; fastest; least accurate
  • Per-channel: one scale per output channel; standard for INT8 weights
  • Per-group (GPTQ, AWQ): groups of 128 weights share a scale; enables INT4 with <1% PPL loss
  • Calibration dataset: 128–512 samples sufficient
Quantisation-Aware Training (QAT)
  • Simulate quantisation during training — fake quantise nodes in forward pass
  • Gradients flow through fake-quant via straight-through estimator (STE)
  • Model learns to be robust to quantisation error
  • Better accuracy than PTQ, especially at INT4
  • QLoRA: 4-bit base model (NF4) + FP16 LoRA adapters — fine-tune 70B on single A100
  • NF4: normally-distributed 4-bit — optimal for normally-distributed weights

Parallelism Strategies for Large Models
distributing compute and memory across accelerators
Data Parallelism (DP)
  • Each device holds a full model copy
  • Different micro-batches processed in parallel
  • Gradient all-reduce after backward pass
  • DDP (PyTorch): overlap all-reduce with backward; efficient at <1B params
  • Scales well; limited by per-device memory (full model must fit)
  • ZeRO (DeepSpeed): shard optimizer states, gradients, params across DP ranks
Tensor Parallelism (TP)
  • Partition weight matrices across devices (column / row sharding)
  • Megatron-LM: column-split Q/K/V projections; row-split output projection
  • Each TP rank computes a partial GEMM; all-reduce/all-gather recombines
  • Requires high-bandwidth interconnect (NVLink) — all-reduce per layer
  • TP=8 typical; beyond 8 communication overhead dominates
  • Sequence parallelism (Ulysses): split along sequence dimension
Pipeline Parallelism (PP)
  • Partition transformer layers across devices: device 0 = layers 0–7, etc.
  • Micro-batching: pipeline bubble reduced by processing B micro-batches simultaneously
  • Interleaved 1F1B schedule: forward-backward alternation; near-zero bubble
  • Communication: P2P activation passing between stages — lower BW requirement than TP
  • GPipe, PipeDream, Interleaved 1F1B — progressive refinements
3D Parallelism — Combining All Three
  • DP × TP × PP: e.g. 512 GPUs = DP=8, TP=8, PP=8 → fits 1T+ parameter models
  • NVLink for TP (high-BW, low-latency); InfiniBand for DP (high-BW); PCIe/NVLink for PP
  • Expert Parallelism (EP): route MoE experts to different devices; all-to-all dispatch/combine
  • GPT-4 / Gemini Ultra estimated: DP+TP+PP+EP combined across >1000 chips

CNN Accelerator Design
convolutional workloads — structure, tiling, and reuse
Convolution Loop Nest
for N, C_out, OH, OW, C_in, KH, KW:
  O[n,c_out,oh,ow] +=
    W[c_out,c_in,kh,kw] ×
    I[n, c_in, oh+kh, ow+kw]
  • 7 nested loops → rich tiling space for data reuse
  • im2col transform: maps conv to GEMM (at cost of memory expansion)
  • Winograd: F(2×2, 3×3) reduces MACs by 2.25× for 3×3 kernels
  • FFT-based conv: efficient for large kernels (>7×7)
  • Depthwise separable (MobileNet): factorises 3×3 → depthwise + pointwise; 8–9× fewer MACs
CNN Accelerator Design Choices
  • Line buffer: store H×W rows of input; slide window without re-reading DRAM
  • Double buffering: DMA fills buffer B while PE array consumes buffer A
  • Weight caching: small kernels (3×3×C_in×C_out) fit in on-chip SRAM; reuse across output tiles
  • Activation reuse: output-stationary on OH×OW tile; stream weights past
  • Fusion: Conv + BN + ReLU fused into single kernel — eliminates 2× DRAM round trips

Power Breakdown in ML ASICs
  • MAC arrays: 40–60% of total power — scale with utilisation; DVFS critical
  • SRAM access: 25–35% — 6T SRAM reads dominate; scratchpad > cache energy
  • DRAM/HBM PHY: 15–25% — major target for compression and KV cache reduction
  • NoC/interconnect: 5–10% — scales with data movement; locality crucial
  • Control & misc: 3–5% — reduced in DSA vs GPU by removing OoO logic
Energy-Reduction Techniques
  • Clock gating: disable MAC array clocks when inactive; 20–40% dynamic power saved
  • Power gating: shut down entire engine island; longer wake latency
  • DVFS: scale V and f to minimum needed; quadratic V reduction
  • Near-threshold computing: V_DD → V_T; 10× energy reduction; area/speed penalty
  • Data compression: compress zero-heavy activations before DRAM transfer
  • Sparsity gating: detect zero activations early; skip MAC; save dynamic power

Emerging Directions
what comes after the current generation
Processing-in-Memory (PIM)
  • Compute inside or adjacent to DRAM die — eliminates bus bottleneck
  • Samsung HBM-PIM: GEMV engine inside each HBM stack; 2× bandwidth for DNN inference
  • SK Hynix AiM: multiply-accumulate units in DRAM row decoders
  • Key challenge: limited compute density; no caches; programming model
  • Best fit: memory-bound ops — embedding lookup, GEMV decode
  • Prototype systems show 2.4× energy improvement for LLM decode
Analog / Resistive Computing
  • Analog in-memory compute (AIMC): weight stored as conductance; dot product via Kirchhoff's law — O(1) energy per MAC
  • Phase-change memory (PCM), ReRAM, MRAM: non-volatile weight storage — zero boot time
  • IBM PCM chip: 512 analogue columns; INT4 equivalent; 85 TOPS/W demonstrated
  • Challenges: non-idealities (drift, noise, write endurance); ADC/DAC overhead consumes gains
  • Likely path: hybrid digital-analog; analog for weight-stationary inference, digital for accum
Photonic & Neuromorphic
  • Photonic MAC: Mach-Zehnder interferometer arrays; perform GEMM at speed of light, zero dynamic power for multiply; Lightmatter Passage chip
  • Neuromorphic: Intel Loihi 2; event-driven spiking neural network; <1 mW for sparse workloads; not competitive with transformers yet
  • Wafer-scale: Cerebras WSE-3; 44 GB SRAM; eliminates HBM for moderate models
  • 3D die stacking: SRAM-on-logic (TSMC SoIC); put 192 MB SRAM directly atop logic die — bandwidth >100 TB/s
  • Timeline: 3D stacking in production now; photonic & analog: 5–10+ years to volume

Summary & Key Takeaways
AI / ML Accelerator Architectures — Presentation 05
Architecture Principles
  • Roofline model guides design: shift ridge point left (more BW) or up (more FLOPS); target workloads in compute-bound regime
  • Systolic arrays maximise data reuse in GEMM; weight/output/row-stationary dataflows trade off different memory resources
  • Dataflow engines (Groq, Cerebras) eliminate runtime scheduling overhead; compiler becomes the hardware
  • Flash Attention: algorithm-hardware co-design moves attention from O(N²) HBM to O(N) — critical for long context
  • Sparsity at multiple levels: activation (ReLU), weight pruning, structured (2:4), MoE
Numerical Formats & Precision
  • Training: BF16/FP16 + FP32 master weights (AMP); FP8 on H100+ with dynamic scaling
  • Inference: INT8 standard; INT4/FP4 for bandwidth-limited decode; FP4 on B200
  • QLoRA/GPTQ/AWQ: PTQ at INT4 with per-group scaling — <1% quality loss
System-Level Themes
  • Memory is the bottleneck — HBM, PagedAttention, KV cache optimisation
  • Parallelism at every level: DP/TP/PP/EP for training; disaggregated serving
  • Compiler stack (MLIR, XLA, TVM, Triton) as competitive differentiator
  • Edge NPUs: 10–50 TOPS/W; UMA avoids bandwidth bottleneck; on-device LLM now real
  • Next frontier: 3D SRAM stacking, PIM, photonic — closing the bandwidth gap
